⚠️ AI 助手必读：核心开发与维护法则
1. 亡灵拦截与数据回滚法则 (Regenerate/Swipe 兼容)
现象： 当用户在酒馆点击“重新生成”或滑动分支时，小手机必须瞬间抹除该楼层产生的历史。
机制： wechat-data.js 拥有 rollbackToFloor(targetIndex)。当 index.js 监听到滑动或重新生成时，会强制删除所有 tavernMessageIndex >= 当前楼层 的消息。
注意： 如果你新增了数据存储模型，必须在 rollbackToFloor 中加入对应的删除逻辑，否则会出现“酒馆回档了但手机消息还在”的幽灵现象。
2. 流式碎片与批次清洗法则 (BatchId)
机制： AI 输出是流式的，正则可能会分段解析出同一条消息。
规则： addMessage 时会携带 batchId。如果同一楼层收到了相同 tavernMessageIndex 且 fromMainChatTag 为真的消息，但 batchId 不同，系统会判定后者是最新输出，前者是流式碎片并予以替换清洗。
3. 跨应用社交联动法则 (GlobalSocialStore)
机制： 蜜语的主播在通过好友申请后，会自动同步到微信通讯录。
规则： 使用 global-social-store.js。蜜语生成的好友申请（含隐藏设定/印象）先存入主库，微信通过 _mergeHoneyPeopleFromGlobalStore 拉取。严禁在 APP 内部直接硬编码读写其他 APP 的私有存储键。
4. 多模态图片代币调包逻辑
机制： 实现“图文无限次穿插”。
规则： AI 发包前，index.js 会识别消息中的 __ST_PHONE_IMAGE_ 占位符。 Fetch 拦截器会在网卡级将其调包为真正的 image_url 多模态数组发送给大模型。修改 AI 发送逻辑时，切勿破坏占位符字符串。
5. 音频/TTS 专属音色保护
机制： 通话功能强制绑定音色，实际发声统一经由 config/tts-manager.js 的 TtsManager.requestTTS。
规则： 微信单聊/通话界面点击播放时，系统会调用 resolveTtsVoiceByName。如果联系人未在通讯录中绑定专属 Voice ID，播放请求将被拦截并提示，以防所有角色都使用同一个全局默认音色。TTS 服务商配置由设置页写入 phone-tts-* 键，新增服务商时必须同步更新 settings-app.js 与 tts-manager.js。设置页 TTS 分区默认折叠并保存展开状态；豆包分区可上传音频复刻、查询训练状态、把训练好的 S_ 音色设为当前音色。豆包复刻可单独配置复刻 Access Token、复刻 APP ID 与 Worker 地址，上传/查询优先请求 /api/clone 与 /api/status，未填 Worker 时才尝试直连豆包接口。豆包官方预置音色通常用 seed-tts-2.0，复刻音色（S_ 开头）需用 seed-icl-2.0，tts-manager.js 会自动纠正这类复刻音色的资源 ID。

📜 AI 指令与标签解析规范
【微信标签 - <wechat>】
包裹： 必须使用 <!-- --> 注释包裹，防止被酒馆 Markdown 渲染破坏。
格式： 支持 ---联系人--- 分隔多人；线下正文注入格式每个联系人块必须包含“接收人：{{user}}”，解析层只同步接收人为 user/酒馆用户名字/小手机微信里用户自己的昵称的消息。
特殊： [红包](金额：100元)、[转账](金额：500元)、[图片]（描述）、[拨打微信语音]。
【蜜语标签 - <Honey>】
结构： 必须包含 ---热门推荐--- 和 ---当前激情直播---。
联动： 包含 [NAI提示词] 供后续生图，好友申请： 供微信继承。
【音乐标签 - <Music>】
结构： [Char|...]、[Meta|...]、[Stats|...]、[Thought|...]、[Replies|...]、[Media|...]、[Likes|...]。
逻辑： [Media] 字段必须提供真实存在的歌曲，系统会自动调用 API 补全播放链接。
【魔坊标签 - 自定义】
机制： 在设置中定义 <包裹>，AI 输出 <包裹>金币: 100</包裹> 时，数据层会自动解析并持久化该状态。

该架构已进化为一套微型操作系统，通过 storage.js 实现数据持久化，通过 api-manager.js 实现独立通信，通过 global-social-store.js 实现跨 App 记忆。修改代码时请优先查看对应的 Data 层 (M) 逻辑，UI 渲染仅负责展示。
